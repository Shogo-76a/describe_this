## 1. サービス概要

- このサービスはどんなサービスか（3行以内）

## 2. このアイデアはどこから生まれたか

### 2-1. きっかけとなった体験・感情

- 英会話レッスンに参加したことがあるが、精神的な緊張が強く、頭が真っ白になって英語そのものを満足に使えなかった。
	- 対人だと緊張してしまう自分には、別の学習方法が必要だと感じた。
- 自分が英語だけで、どれくらい正確にイメージを相手へ伝えれるのか確認したくなった。
- Gemini などの AI を使えば画像生成と採点はできるが、下記の問題があった。
	- 毎回ルールを説明する必要がある
	- 画像の比較が面倒
	- 採点基準が曖昧
- 「この機能に特化したアプリがあれば、もっと手軽に練習できるはず」と探してみたが、ちょうどいい製品が見当たらなかった。

せっかくなので、この際に作ろうと思った。

## 8. このアプリの懸念点とその対策
### 懸念点①

- 何が問題になりそうか：
	- 採点結果に効果があるフィードバックをできるかどうか。
	- 同じお題で次回プレイ時にフィードバックを反映すると、得点が前回よりも上がるのかどうか。
-  なぜそう思うか：
	- 構成が、「GPT OpenAI（採点） と Flux BlackForestLab（画像生成）」というように別メーカーのAIモデルを採用する予定なので、フィードバックと画像の生成結果で整合性が取れるかどうかが分からない。言葉のつながりを見つける思考パターンが違うのかなという心配がある。
	- 感受性が違う2人の人物に同じ言葉を使っても、理解する内容が違うということが起きるのかなと思う。「犬」という単語に対して片方は「柴犬」を、もう片方は「トイプードル」を想像するみたいな感じで。
- そのために考えている対策：
	- GPT に説明文を理解させ、GPT が Flux に命令文を記述して送るという流れを考えている。
	- そうすることで、一つのモデルだけがユーザーの説明を理解して、その理解をもとに指示を出し、結果を採点できるようになる。GPTを脳、Fluxを手という関係にすることで、フィードバックと生成結果に整合性が出てくるのではないかと考えている。

		- 今回の用途では、生成画像の品質よりも、速度とコストを重視している。
	- GPT-4o-mini
		- カリキュラムで使ったことがあり、導入が容易で料金が安い。
		  今回求める品質には十分そう。
- API
	- DeepInfra
		- FLUX.1 Schnell をAPIとして利用するために採用。
		  OpenAIのAPIと互換性があり、OpenAIが導入済みであれば、URLとAPIキーの書き換えだけで利用できる。
	- OpenAI
		- カリキュラムで使ったことがあり、導入が容易。
	- Lorem Picsum
		- 版権フリー画像を利用できる。
	- Web Speech API（MVPではなく、本リリースで実装）
		- ChromeやSafariなど、主要なブラウザに標準搭載されている。
		- 無料で利用できる。
- 使用予定ライブラリ
	- ruby-openai
		- AIモデルを使うため。
	- annotate_models
		- データテーブルの構造を関連ファイルに記述。把握しやすくする。
	- rails_admin
		- 管理画面作成ライブラリ。設定の手間が少なく、初心者に優しそう。
	- Webmock
		- テスト時、APIなどの外部通信を原則禁止するため。
		- または、簡素なモックデータが必要なときに利用する。
	- VCR
		- テスト時、API部分の処理を一度記録/スタブ化する。

### 10-2. 技術選定の理由
- 今回チャレンジしたい点
	- ユーザーの待ち時間がなるべく発生しないように、HotwireやSolidStackを使ってページの更新または動作が止まらないアプリを作りたい。
	- daisyUIは今回初めて使うので、公式ページを見つつ使い方に慣れたい。
	- スマホでも使う事を考えてるので、スマホ上での見た目と使いやすさに気を付けていきたい。PWAで実装進める。
- 不安な点
	- スマホでも問題なく動作させられるかどうか。
	- 工数見積もりをこれからしていくが、知らない技術の調査を含めたときに、卒業期限に間に合うかどうか。工数通りにいかなかった場合は、いくつか実装を見送るかもしれない。
